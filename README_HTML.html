1 Teoría

En este paquete diseñado para R, se ha implementado una regresión logística para clasificar una variable con dos clases (por ejemplo: 0/No-1/Sí). Para ello, se utiliza la función sigmoide, que viene definida por la siguiente ecuación: \[h(z) = \frac{1}{1+\exp{(-z)}}\]

donde: \[ z = \theta^{T}\cdot x\]

La salida de esta función siempre estará contenida entre 0 y 1, como podemos observar en la siguiente imagen:

Función Sigmoide

En la función sigmoide, \(\theta\) representa los pesos asignados. Definimos una función de coste que se minimizará para obtener los valores óptimos de los pesos, \(J(\theta)\): \[ J(\theta) = -\frac{1}{m}\sum_{i=1}^{m}{[y_{i}\cdot log(h(z_{i})) + (1-y_{i}) \cdot log(1-h(z_{i}))]}\]

Utilizando el algoritmo de descenso del gradiente, y a través de la derivada parcial, obtenemos los valores óptimos: \[\frac{\partial{J(\theta)}}{\partial{\theta_{j}}} = \frac{1}{m} \sum_{i=1}^{m}{[h(z^{i})-y^{i}] \cdot x_{j}^{i}}\]

Los pesos se van actualizando a partir de la siguiente ecuación: \[\theta_{i} = \theta_{i-1} - \alpha \frac{\partial{J(\theta_{i-1})}}{\partial{\theta_{i-1}}}\]

donde:

  • \(\alpha\) = ratio de aprendizaje

En nuestro paquete, el algoritmo de descenso del gradiente se frenará en base a dos parámetros: el número máximo de iteraciones (max_iter) o cuando se alcance un valor umbral (threshold):

  1. El algoritmo se parará cuando se alcance el número máximo de iteraciones.
  2. Si se define un valor umbral, el algoritmo se frenará cuando \(h(z) < umbral\), independientemente del número de la iteración.

2 Función

La función logistic_regression creada sigue la siguiente lógica:

  1. Llama a la función logistic_reg_comprobations donde se comprueba que los parámetros de la llamada a la función sean correctos.

  2. Llama a la función prepare_logistic_reg_data en el que se hace una multiasignación para dividir los datos de entrada en x_train, y_train, x_test y theta.

  3. Llama a la función gradient_descent para obtener los valores óptimos de theta (los cuales minimizan la función de coste). También se calcula el valor de la función de coste para cada iteración.

  4. Llama a la función predict en la que se predicen los valores tanto para la variable target del dataset de entrenamiento como el de test.

  5. Análisis de los resultados (matriz de confusión, gráfico de dicha matriz, etc…)

3 Argumentos

Los argumentos de la función logistic_regression son los siguientes:

## function (train, test, target, lr = 0.001, max_iter = 1e+05, 
##     probs = F, threshold = NULL, regu_factor = NULL, regu_method = "ridge", 
##     epsilon = 1e-05, ...) 
## NULL

donde:

  • train: Dataset de entrenamiento
  • test: Dataset de test
  • target: variable que se ha de predecir (valor numérico)
  • lr: Learning Rate para el algoritmo del descenso del gradiente
  • max_iter: Número máximo de iteraciones en el algoritmo del descenso del gradiente
  • probs: F si los resultados son categóricos, T si los resultados se dan como probabilidades (entre 0 y 1)
  • regu_method: “ridge”, regularización de Ridge
  • regu_factor: Factor de regularización para el método Ridge. Si no se cubre, no se usa regularización.

Esto nos permite obtener ya nuestras primeras conclusiones. Parece que el valor “?” actúa en este dataset como NA, por lo que habrá que analizar sus implicaciones posteriormente. Por otro lado, en base a los datos, parece claro que el objetivo de este problema será predecir si una persona va a tener unos ingresos superiores a 50k o no. Sobre las variables ‘sex’ e ‘income’ se aplicará el proceso one-hot encoding, que consiste en transformar variables categóricas en numéricas, haciendo más ancho el data.frame (por cada nivel de la variable se creará una columna). En este caso, como las variables sólo tienen una columna, podemos almacenar toda la información en una sóla columna (en un one-hot encoding, las columnas necesarias serán n-1 siendo n el número de niveles). Por otro lado, también es necesario comentar que sobre estas variables podría realizarse una discretización de las mismas, pero no se ha hecho por los siguientes motivos: hay muchos algoritmos que aceptan valores numéricos pero no factores (por ejemplo, las redes neuronales) y por otro lado, trabajar con valores numéricos siempre es más eficiente que con otro tipo de datos.

4.4 Función de Preprocesamiento

Una de las librerías más importantes en R que permite hacer preprocesamiento de datos es ‘recipes’. Se ha creado una función basada en esta librería que facilita la aplicación de todas las funciones de preprocesamiento que contiene sobre variables numéricas o categóricas para variables que actúan como predictoras o cómo variables de salida. Esta función fue originalmente creada para una librería que he creado en la que se implementaba el algoritmo k-Nearest Neighbours (knnClassifier library):

#' @title Preprocess
#' @description This function applies step functions of recipes package
#' @author Alberto Almuiña
#' @param formula Formula that will be evaluated in algorithm (for example: Species ~ .)
#' @param data Dataframe that will be modified
#' @param methods Steps functions of the recipes package to be applied. For example( c("step_center", "step_scale", "step_range"))
#' @param not_apply_on "nominal" or "numeric". The column type in which step functions will not be applied. 
#' @param role "predictor" or "outcome". Outcome is the target variable and "Predictor" represent all the rest in "formula".
#' @return
#' Returns a data frame with all the changes applied.
#' @export
#' @examples
#' data<-preprocess(Species ~ ., iris, c("step_center", "step_scale", "step_range"), "numeric", "predictor")
#' data<-preprocess(Species ~ ., iris, c("step_dummy"), "nomianl", "outcome")


preprocess<-function(formula, data, methods = is.character(), not_apply_on = c("numeric", "nominal"),
                         role = c("predictor","outcome"), ...){
        
        methods<-match.arg(methods, 
                           ls("package:recipes"),
                           several.ok = TRUE)
        
        not_apply_on<-match.arg(not_apply_on)
        
        role<-match.arg(role)
        
        recipes<-recipe(formula, data = data)

        step_functions<-map(methods, match.fun)

        data<-map(1:length(step_functions), function(x){

                prep<-recipes %>% step_functions[[x]](has_role(role), -has_type(not_apply_on)) %>% prep()

                bake(prep, data)

        }) %>% .[[length(step_functions)]] %>% as.data.frame()
        
        return(data)
}

Vamos a reescalar las variables numéricas para que todas se encuentren en el rango (0-1), seleccionar las variables que consideremos de interés para nuestro modelo (en base a lo que se fue comentando anteriormente), así como transformar en variables dummies aquellas variables categóricas que necesitan ser pasadas al algoritmo.

datosAdult <- preprocess(income ~ ., 
                         data = datosAdult,

create_data_partition<-function(data, p_train){
        
        sampled_data<-partial(sample, size = round(p_train*dim(data)[1]))
        
        index<-do.call(sampled_data, list(1:dim(data)[1]))
        
        train<-data[index,]; test<-data[-index,]
        
        return(list(train = train, test = test))
}

c(train, test) %<-% create_data_partition(datosAdult, 0.8)

4.7 Conclusiones

En cuánto a las variables:

  • Cuánto mayor es el nivel educativo de las personas, mayor es el número de personas que ganan más de 50k. Además, hay dos variables que nos aportan exactamente la misma información, por lo que se prescinde de ‘education’ y se mantiene la variable numérica (más fácil de procesar para el algoritmo).

  • Hay más hombres que mujeres que ganan más de 50k.

  • Pueden observarse importantes diferencias de ganancias entre las diferentes profesiones y también entre los distintos sectores laborales (aunque estas diferencias son menores cuando se agrupan los niveles en Gobierno y Auto-Empleado).

  • Las personas menores de 30 años tienen unos ingresos mucho menores debido posiblemente a que aún están empezando su carrera profesional. Sin embargo, los dos siguientes niveles (de 30 a 55 y de 55 a 100 años) son muy similares.

  • Casi un 50% de los casados ganan más de 50k, estando todos los siguientes niveles en torno al 10%. La misma información puede obtenerse al analizar la variable ‘relationship’, donde casi el 50% de los maridos y mujeres supera el umbral de los 50k. Por este motivo, al aportar casi la misma información, se descarta la variable ‘relationship’.

  • En cuánto a la raza, el dataset está muy desbalanceado en favor de la clase ‘White’. Aunque pueden apreciarse ciertas diferencias entre las clases, habría que intentar igualar más las clases.

  • El 89% de las personas son de Estados Unidos, mientras que el 11% restante está muy dividido en múltiples clases. Esta variable no es de utilidad para la predicción del ‘income’ (se han hecho pruebas incluyéndola y el resultado siempre disminuye, añadiendo ruido al modelo).

  • Puede observarse cómo las mujeres que ganan más de 50k trabajan más horas a la semana, al igual que sucede con los hombres (aunque la diferencia no es tan notable respecto a los que no lo ganan). Por otro lado, si se analizan las horas trabajadas por sexo y nivel educativo, también se observa como para las personas con estudios avanzados se trabajan más horas.

  • A partir de las ganancias y pérdidas de capital, puede obtenerse la variable ‘ganancias’, que es la resta de ambas. A pesar de que el 87% de las personas no tienen ninguna ganancia, esta variable es importante para el modelo porque ayuda a distinguir a las personas que sí ganan más de 50k.

En cuánto al preprocesamiento de los datos:

  • Se han realizado agrupaciones de niveles para las variables ‘workclass’, ‘education’ y ‘marital_status’. Por otro lado, se ha agrupado la edad y el nivel educativo.

  • Se han convertido los valores ‘?’ en NA tanto para workclass como para occupation. Posteriormente, se han imputado estos valores mediante el algoritmo k-Nearest Neighbour, buscando la información entre los vecinos más próximos.

  • Se han reescalado las variables numéricas que se han utilizado en la predicción entre 0 y 1 debido a las grandes diferencias entre sus valores.

  • Las variables categóricas se han transformado en variables ‘dummy’ para poder ser procesadas por el algoritmo.

  • Variables con muy poca correlación entre ellas (exceptuando los distintos niveles de las variables dummy entre ellas)

En cuánto al algoritmo usado para predecir:

  • Se ha utilizado la función logistic_regression de la librería LogisticRegression creada por el autor de este trabajo, que permite la clasificación binaria.

  • Se han obtenido unos resultados muy decentes (teniendo en cuenta que no se ha hecho ningún tipo de hyper-parameter tuning). El principal problema encontrado está en el desbalanceamiento de la clase ‘income’ y que la regresión tiende a predecir ‘0’, por lo que obtiene unos valores para la ‘specificity’ bastante bajos.

  • Para intentar solucionar el problema anterior, se utiliza la regularización Ridge para penalizar al algoritmo, consiguiendo aumentarse el número de clasificaciones ‘1’ hechas correctamente (aumentando la specificity).

  • También se comprueba que la variable ‘ganancias’ creada mejora en torno a 1% el resultado.

  • Finalmente, se ha realizado un análisis de componentes principales para descartar 10 dimensiones y obtener un resultado bastante similar al inicial (técnica importante cuando el coste computacional es muy importante). Se utilizan las primeras 23 componentes que explican el 98% de la varianza de los datos.
